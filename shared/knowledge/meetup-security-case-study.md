## 根因分析

| 问题 | 说明 |
|------|------|
| **RLS 未启用** | Supabase 表默认无行级安全 |
| **匿名 key 可读写** | 前端直接调用，无需认证 |
| **开发阶段遗留** | 快速原型时忽略安全配置 |

**关键误区：**
> "这只是个报名表，没什么敏感数据"

实际上：姓名 + 微信 = 可直接联系真人 = 社交工程入口

---

## 修复措施

1. **启用 RLS** — 表级别强制策略
2. **匿名用户拒绝** — `401 Unauthorized`
3. **授权方式** — 需要登录/token 才能读写

**验证：**
```bash
# 修复前：200 OK + 44 条数据
curl .../neonclaw_signups  # → 200

# 修复后：401 Unauthorized
curl .../neonclaw_signups  # → 401
```

---

## 核心教训

### 给开发者的 5 条原则

1. **RLS 不是可选项** — Supabase 表必须显式启用
2. **匿名 key ≠ 安全** — 默认拒绝，显式授权
3. **PII 最小化** — 只收集必要信息
4. **安全审计常态化** — 上线前检查表权限
5. **快速响应** — 发现 → 报告 → 修复，90 分钟内闭环

### 给活动组织者的建议

- 使用成熟平台（而非自研表单）处理敏感报名
- 如必须用 Supabase，启用 RLS 后再部署
- 定期审计：检查哪些表暴露了数据
